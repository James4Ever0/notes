# Full home server management system

Remote shutdown, remote sleep, remote restart, remote management (kvm)

Only power up system when needed. Use hardware along with Wake on Lan bios feature.

Operate comouter remotely via lora or ypn tunnels

---

You can definitely set up a Linux system to hibernate and wake up remotely over a network. It involves two key parts: the ssh command to trigger hibernation and the network standard Wake-on-LAN (WoL) for powering it back on.

🧠 How It Works

Here's a quick breakdown of the process:

· Hibernate (Going to Sleep): You connect from your client machine to the Linux server via ssh and run a command like sudo systemctl hibernate. This safely saves the system's state to disk and powers it down.
· Wake (Waking Up): Your client sends a "Magic Packet" over the network to the sleeping server. The server's network card, which remains in a low-power state, detects this packet and triggers the system to boot back up.

⚙️ Step-by-Step Configuration

Part 1: Remotely Hibernate via SSH

1. Enable SSH on Your Linux Server:
   · Install the SSH server if you haven't already. On Debian/Ubuntu, use sudo apt install openssh-server.
   · Ensure the sshd service is running and enabled (sudo systemctl enable --now sshd).
2. Set Up SSH Key Authentication (Recommended): To log in without a password, generate a key pair (ssh-keygen) on your client and copy the public key to the server (ssh-copy-id user@server_ip).
3. Test the Hibernate Command: From your client, run ssh user@server_ip 'sudo systemctl hibernate'. You might need to configure sudo for passwordless execution of systemctl hibernate by editing /etc/sudoers with sudo visudo and adding a line like: user ALL=(ALL) NOPASSWD: /usr/bin/systemctl hibernate.

Part 2: Remotely Wake via Wake-on-LAN (WoL)

1. Hardware Prerequisites: Your computer's motherboard, BIOS/UEFI, and network card (NIC) must all support WoL. Most modern systems do.
2. Enable WoL in BIOS/UEFI:
   · Reboot and enter the BIOS/UEFI setup (usually by pressing Del, F2, or F12 during boot).
   · Find the power management settings. Enable options named "Wake-On-LAN," "Power On By PCI-E Device," or similar.
3. Enable WoL in Linux OS:
   · Install the ethtool utility (sudo apt install ethtool).
   · Identify your network interface name with the ip a command (e.g., eth0 or enp3s0).
   · Check current WoL status: sudo ethtool <interface_name> | grep Wake-on. A value of g indicates it's enabled.
   · If not enabled, activate it: sudo ethtool -s <interface_name> wol g.
   · Make the setting persistent across reboots: The WoL setting is lost on reboot, so you'll need to make it permanent. You can use a systemd service or a cron job.
4. Send the Wake-up Command:
   · On your client machine, install a tool like wakeonlan (sudo apt install wakeonlan) or etherwake (sudo apt install etherwake).
   · Note the MAC address of the sleeping server's network interface (ip a).
   · Send the magic packet from your client: wakeonlan <MAC_ADDRESS>

🌍 Waking Up Over the Internet (WAN)

Waking a computer over the internet is more complex, as WoL is designed for local networks. Here are a few solutions:

· VPN (Recommended): Connect to your home network via a VPN first, then send a standard WoL packet. This is secure and requires no special WoL handling.
· Port Forwarding (Less Secure): Configure your router to forward a specific UDP port (usually 7 or 9) to the broadcast address of your LAN. However, many routers don't allow forwarding to a broadcast address, and this method poses a security risk.
· Dedicated Low-Power Device: Use a device like a Raspberry Pi that stays on 24/7. Securely connect to it from the internet and use it to send the WoL packet.

🐛 Troubleshooting Tips

· Hibernation Command Fails: Ensure you have a swap partition or file that is at least as large as your RAM. Run sudo systemctl hibernate locally on the server to confirm it works.
· WoL Doesn't Work: After running ethtool, double-check that the Wake-on line shows g. Also, try using etherwake as an alternative tool.
· System Wakes Up Immediately: This is a known bug where the network card receives a false WoL signal. A workaround is to disable WoL just before hibernation and re-enable it after. You can automate this with a script in /etc/systemd/system/.

I hope this guide helps you set up your remote power management. If you run into any issues with specific error messages or your hardware setup, feel free to ask for more help.

You don't need to rewire any power pins; Wake-on-LAN (WoL) is specifically designed to start a computer over the network without any physical modification. However, whether WoL works from a full shutdown (S5 state) depends entirely on your specific hardware and its configuration.

🚫 Do You Need to Rewire Power Pins?

No. While it is technically possible to start a computer remotely by wiring a relay or microcontroller (like an Arduino) to the motherboard's power button pins, this is a complex, DIY hardware project. For a standard software-based solution, WoL is the correct and only method needed.

🛌 Does Wake-on-LAN Work After a Full Shutdown (S5)?

This is the critical nuance. The ACPI "Soft Off" (S5) state is the full shutdown you initiate via your OS. In this state, the main components of the computer are unpowered.

· How it's supposed to work: The network card must stay on a "wake-up armed" mode using a small amount of standby power from the power supply. When it receives a special "Magic Packet," it signals the motherboard to power on the entire system.
· The Reality: This is not always guaranteed. Many users report that WoL works flawlessly from sleep (S3) but fails from a full shutdown (S5). This can be due to several factors:
  · BIOS/UEFI Settings: The option might be named "Wake on PCIe," "Power On by PCI-E," or "Wake on LAN." Ensure it's enabled.
  · Operating System Configuration: On Linux, you must set WoL with ethtool and ensure this setting persists across reboots.
  · Network Driver Issues: A bug in the network driver may power down the interface before the WoL system is armed, breaking the feature.
  · Hardware Limitations: Some network chipsets simply do not support WoL from S5 at all.

In short, you never need to rewire hardware pins for remote startup. WoL is the software standard that handles this, but its reliability from a full shutdown varies and is highly dependent on your specific setup.
