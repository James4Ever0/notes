# DSL in agentic development coding

你提的这个方向非常前沿，已经踩到了**AI辅助软件工程的核心痛点**：如何让代码不再是唯一真相源，而是由可审计、可分派、可跨模型协作的意图规格驱动。你设想的这套体系，市面上并没有一个完全同名的现成项目叫“代码审计中间语言”，但确实存在一组相互独立但可以组合的技术和思想，正好能拼出你想要的东西。

下面我按你的提问拆开来说，并给出已有的 GitHub 项目、术语和架构参考。

---

## 1. 这类语言有没有标准名称？

没有标准叫“代码审计中间语言”，常见的相关术语是：

- **意图规格语言（Intent Specification Language）**  
  比如你描述的 Pev，本质就是一种意图规格。
- **可执行规范（Executable Specification）**  
  如 Gherkin（BDD 的 `Given/When/Then`），但偏测试，不涵盖任务分派、边界约束。
- **模型驱动工程里的领域特定语言（DSL, Domain-Specific Language）**  
  如 UML 的 XMI、SysML，但太“重型”且不面向 AI 可控生成。
- **架构描述语言（ADL）**  
  如 AADL、Darwin，偏组件和连接器，不直接包含意图激活/版本/黑白名单。
- **AI 工作流描述语言**  
  比如 LangChain 的 `Chain` 定义、Dify 的可视化管道、AutoGen 的 agent 对话配置等，更接近任务分派，但缺乏“代码投影”和双向审计。

你的 Pev 实际上融合了：
- 意图版本管理
- 功能激活标记
- 约束边界（黑白名单）
- 模块/子任务分派
- 生成与审计闭环

所以可以给这类 DSL 一个新名字，例如 **Intent‑to‑Code DSL** 或 **Auditable Intent Specification**。

---

## 2. 能否跨语言解析？类似 Tree‑sitter 那种？

**完全可以，而且 Tree‑sitter 正是最适合做这件事的工具之一。**

- Tree‑sitter 可以定义任何 DSL 的语法，生成 C 解析器，并提供 Rust、JS、Python、Go 等多种语言的绑定。
- 你可以用 Tree‑sitter 实现 Pev 的解析器，做到：
  - 语法高亮
  - AST 查询（类似代码审计）
  - 跨语言生成代码框架（不同目标语言的代码模板）
- 类似的工具还有 ANTLR（生成多语言 parser）、Langium（TypeScript DSL 框架）、TextX（Python DSL 框架）。但 Tree‑sitter 生态最广，适合嵌入到编辑器、CI、代码生成器里。

所以：**你完全可以为 Pev 写一个 tree‑sitter‑pev 语法，然后基于它做跨语言解析和代码生成。**

## 3. 有没有现成的类似 harness 框架或项目？

我分几类给你列出 GitHub 上已有的“脚手架”项目，虽然不完全一样，但可以组合起来实现你的想法。

### 3.1 多 Agent 任务分派与编排（可以给小模型分活，大模型干细活）

- **Microsoft AutoGen**  
  多智能体对话框架，你可以定义不同角色（Planner, Coder, Reviewer），用自然语言或结构化配置分派任务。  
  - 适合：大模型写代码，小模型生成计划/审核。
  - 可以用 JSON/YAML 定义 agent 行为，相当于一种轻量 DSL。
- **CrewAI**  
  角色基的多 Agent 协作，同样可以按能力分派。
- **MetaGPT**  
  用 SOP 模拟软件公司，产出 PRD、设计、代码。内部已有类似意图传递的文档流。
- **ChatDev**  
  类似，用提示词扮演不同角色。

这些框架都有一个**任务描述协议**，通常是自然语言或半结构化提示，你可以把它们的一层封装成 Pev DSL。

### 3.2 意图/规范驱动代码生成（先写规范，再生成代码，再反过来校验）

- **GPT‑Engineer**  
  在 `gpt-engineer` 项目中，有一个 `spec` 步骤，你在生成代码前先写一个 `spec.md`（人类可读），然后 AI 按规范生成代码。  
  - 可以认为这个 `spec.md` 就是简易 Pev。
- **Smol‑ai/developer**  
  也是先写需求文档，再生成代码。
- **Aider**  
  交互式结对编程，有 `/architect` 模式先定设计，再写代码。它有一个“编辑规格”概念。
- **Plandex**  
  用“计划”驱动长期任务，有明确的上下文和边界管理。

### 3.3 用于审计和强制约束的 DSL / 规则语言

- **Semgrep**  
  支持跨语言代码模式匹配，可用来**检查生成的代码是否违反 Pev 中设定的边界**（例如不允许直接操作数据库、不能调用某 API 等）。
- **CodeQL**  
  更强大的语义查询，可以写安全规则。你可以从 Pev 生成 CodeQL 查询，验证代码实现是否符合意图。
- **CUE / Pkl**  
  数据约束语言，擅长定义配置和 schema，可用来描述 API 边界、模块间契约。这些很容易作为 Pev 的子集。

### 3.4 模板与框架生成

- **Cookiecutter** 或 **Yeoman**  
  根据模板生成项目骨架，可配合 DSL 控制生成内容。
- **Tree‑sitter + codegen**  
  很多项目用 tree‑sitter 解析 DSL，再驱动代码生成（如 `smithy` 生成多语言 SDK）。

## 4. 怎么实现你描述的 Pev 闭环？（架构设想）

基于现有工具，可以这样组合实现你的 Pev 方案：

```
Pev DSL 定义 (Tree-sitter 语法)
        ↕ 解析/生成
Pev 文件 → [小模型] 生成/修正 Pev（意图级别）
        ↕
[Harness 框架] 解析 Pev，拆解任务与边界条件
        ↕
多 Agent 分发：
  - Agent A (大模型) 根据 Pev 子任务生成具体代码
  - Agent B (小模型) 审核代码是否符合 Pev 约束（通过生成 Semgrep/CodeQL 规则）
        ↕
生成代码 → CI 检查（用 Pev 生成的规则）→ 若失败，反馈修正 Pev 或代码
        ↕
Pev 版本与代码版本关联，diff Pev 即可解释业务变化
```

### 关键组件选型建议：
- **Pev 语法**：用 Tree‑sitter 自定义，方便集成到 VSCode、Neovim、CI。
- **任务分发**：用 AutoGen 或 CrewAI 构建多 Agent 流水线。
- **约束检查**：从 Pev 中提取“不可逾越边界”自动生成 Semgrep 规则。
- **小模型可读性**：Pev 语法可设计得极简，甚至像 YAML 或你草案中的 `prefix keyword [attr] : body` 风格，方便 7B 模型理解。
- **跨语言代码生成**：为每种目标语言准备 Jinja2/Liquid 模板，由 Pev AST 驱动渲染。

## 5. 这种方案能否“拓展模型边界，干不了的活交给别的模型”？

**完全可以。**  
这正是多 Agent 编排的强项。例如：

- 小模型（如 Phi‑3）因为 Pev 形式简单，可以很可靠地**生成和修改 Pev 文件**（意图规划）。
- 大模型（GPT‑4/Claude）专注**把 Pev 子任务变成高质量代码**。
- 如果大模型写出的代码不符合约束，**另一个模型（或静态检查工具）直接拒绝**，并生成反馈消息让编排器重新修正 Pev 或代码。
- 一些纯形式化检查（如类型边界、API 黑白名单）根本不需要模型，直接用 Semgrep 就能挡掉，从而“不浪费模型能力”。

这种架构天然形成了 **“可信内核（DSL+工具）+ 大模型外围”** 的模式，模型只是执行者，**规范永远留在 Pev 里**。

## 6. 你草案里的 Pev 思路评价

你提出的 Pev 结构（`@--{...}` 装饰器、`match mode`、`%X`、优先级规则等）**已经是一种雏形 DSL**。它的价值在于：

- 把意图和代码剥离，让代码变成投影 → 颠覆“代码即真相源”。
- 版本 diff 可解释业务变化 → 非常适合 AI 参与的迭代。
- 注释和 `:?` 修订模式 → 直接在提示词框架内实现迭代改进。

完全可以把它形式化，用 Tree‑sitter 定义语法，然后按上述架构实施。

你描述的这套构想——用一套独立的规范语言作为“真实来源”，驱动AI生成代码、校验产出、分配任务——是一个非常前沿且系统化的设计。你提到的“Pev”与当前业界多个热门方向不谋而合。

目前，这个领域还没有一个统一的、被广泛接受的官方术语。但你的构想恰好处于几个核心概念的交叉点上：

*   **规范驱动开发 (Specification-Driven Development, SDD)**：这是与你想法最吻合的顶层方法论。其核心正是“以规范为可执行契约”，将传统文档转化为具有强制约束力的自动化校验关卡。你的Pev就是这个“规范”。
*   **意图规范语言 (Intent Specification Language)**：这更贴近Pev的本质——描述“做什么”而非“怎么做”。例如，`application-intent-model (AIM)` 项目就是“一种意图驱动的规范语言”。
*   **代码作为投影 (Code as Projection)**：这是Pev最核心的理念，即代码不再是唯一真实来源，而是Pev意图在特定技术栈下的一个“投影”。这种“双向同步”的思想在OpenSPDD等项目中被明确强调。

### 🧩 现有类似项目与概念

你构想的“Pev”并非孤例，业界已有多个项目在不同层面实现了类似思想，可作为重要参考：

*   **OpenSPDD**：一个“结构化提示驱动开发”的CLI工具。它通过“REASONS Canvas”框架从7个维度（需求、实体、方案、结构、操作、规范、防护）来结构化设计，并支持设计与代码的**双向同步**。
*   **Application Intent Model (AIM)**：一种意图驱动的规范语言。它将产品行为捕捉在结构化的`.aim`文件中，并引入“意图↔代码绑定层”进行**漂移检测**。
*   **ai-spec**：一个AI驱动的开发编排工具。它能够将一行需求转化为生产就绪的代码，并提炼出“人可读的Spec + 机器可读的DSL”的双层合约。
*   **Stim**：一个用于编写AI提示词的DSL。它将提示词视为源代码，支持变量、控制流和包管理等编程语言特性。

### 🏗️ 跨语言解析与Tree-sitter

你提到的跨语言解析和Tree-sitter，是实现你构想的关键技术基础。

*   **可行性**：完全可行。通过解析代码的**抽象语法树（AST）**，可以提取出函数、类、变量等结构化信息，与Pev中的规范进行比对。
*   **Tree-sitter的作用**：`Tree-sitter`是一个高性能的增量解析库，可为多种编程语言生成AST。已有项目利用它构建**AI原生编程语言**，或进行**多语言代码检查**。它非常适合作为你系统中“代码→Pev”反向同步的解析引擎。

### 🔧 Harness框架体系

“Harness”这个词在AI编程领域确实指代一类框架，与你设想的“任务分配”和“边界约束”高度相关。

*   **Harness框架**：这是一种“结构化预配置”框架。它通过在AI执行任务前提供**架构蓝图、领域知识、角色定义和流程编排**来提升输出质量，并将质量从49.5分提升至79.3分。
*   **Harness DSL**：一些项目（如AWS的OMA）提供了更具体的`Harness DSL`。它允许开发者以声明式YAML文件定义插件和策略，并在工具执行前进行拦截和校验，实现**强制性策略执行**。
